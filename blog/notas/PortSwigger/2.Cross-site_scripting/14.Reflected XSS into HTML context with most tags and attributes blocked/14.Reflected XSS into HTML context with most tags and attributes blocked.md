## LAB

![[20260329161620.png]](20260329161620.png)

En el laboratorio vemos que al insertar un tag como `script`, este no permite realizar la request.

![[20260329161656.png]](20260329161656.png)

Por lo que se realizará la búsqueda de los tags que si están permitidos, esto con ayuda del siguiente recurso en el que encontraremos un listado para xss.

- https://portswigger.net/web-security/cross-site-scripting/cheat-sheet

![[20260329161805.png]](20260329161805.png)

Para ello usaremos el intruder de burpsuite

![[20260329162036.png]](20260329162036.png)

Copiaremos todos los tags para ir iterando y ver cual es la etiqueta el que no esta bloqueado.

![[20260329162105.png]](20260329162105.png)

Al  enviar las solicitudes, veremos que existe una etiqueta el cual no esta siendo bloqueado. por lo podemos usar este etiqueta.

![[20260329162130.png]](20260329162130.png)

Luego tenemos que encontrar algún evento que pueda ocasionar un xss, por lo que también usando la lista de payloads de eventos del recursos de portswigger iteraremos.

![[20260329162550.png]](20260329162550.png)

Vemos que el evento que se usa es `onrisize`

![[20260329163708.png]](20260329163708.png)

Para ello podemos hacer uso de una extensión en el buprsuite y ver que evento puede ejecutar:

![[20260329163730.png]](20260329163730.png)

Por lo que al realizar la solicitud con lo siguiente vemos que este despliega un xss de impresión 

```c
?search=<body onresize=print()>
```

![[20260329163905.png]](20260329163905.png)

Para enviarselo a la vicitima haremos uso del exploit server.

```c
<iframe src="https://0a6c00cb0388d07d81081bb200fe00e3.web-security-academy.net/?search=<body onresize=print()>"  onload=this.style.width="100px">
```

Usando `onload` realizaremos que el iframe crezca en un 100px automáticamente cuando la victima ingrese, luego solo daremos a entregar a la victima para solucionar el laboratorio.
